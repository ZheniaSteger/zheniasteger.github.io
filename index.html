// about

Engineering security in, from day one

I'm a software engineer who grew into a DevSecOps leadership role. My day-to-day sits at the intersection of development, security, and operations - designing the systems and the guardrails that let teams deliver software that's trustworthy by default.

I care about the unglamorous things that make engineering organizations resilient: reproducible builds, signed artifacts, least-privilege everything, fast feedback, and incident response that doesn't rely on heroics. I believe the best security is the kind developers barely notice because it's built into the path of least resistance.

I still write code daily - mostly C++ and Python - review architecture, and keep my hands on the tools. I'm happiest turning a fragile, manual process into something automated, observable, and boring.

Right now I'm focused on the next frontier: bringing DevSecOps discipline to AI and machine learning. Securing model and data pipelines, hardening the infrastructure that serves them, and treating ML systems with the same rigor I'd apply to any critical workload.

// how i work

Principles I lead by

01

Shift left, shield right

Catch issues at the keyboard, not in production - but assume breach and design defense in depth anyway.

02

Automate the critical path

If a control is manual, it's optional. Security that scales is security that's codified and enforced by the pipeline.

03

Make the secure way the easy way

Developers ship faster when guardrails are paved roads, not roadblocks. Friction is a vulnerability.

04

Measure, don't assume

Observability, SBOMs, and audit trails turn "we think we're secure" into evidence you can stand behind.

05

Trust the supply chain you built

A system is only as trustworthy as its weakest dependency. Sign what you ship, verify what you pull, and know exactly what's in production.

06

Security is a team sport

Guardrails stick when engineers own them. I'd rather coach a team to ship securely than be the gate everyone learns to route around.
